# Composer Classification

Classifies MIDI files by composer (Bach, Beethoven, Chopin, Mozart) with a hybrid
CNN and LSTM model in PyTorch. Each song enters the model twice: as a piano roll
that the CNN reads like an image, and as a vector of 37 handcrafted music theory
features (melodic intervals, vertical intervals, Krumhansl key fit, rhythm on the
beat grid) fused at the dense head.

The final model scores **macro-F1 0.812 +/- 0.021** under stratified 5 fold cross
validation of 1,628 songs. Bach is nearly solved (97% recall); the remaining errors
concentrate in the Beethoven vs Mozart pair.

## Data

The MIDI files come from the Kaggle
[MIDI Classic Music](https://www.kaggle.com/datasets/blanderbuss/midi-classic-music)
archive. Download and unpack it so the composer folders sit at
`data/raw/archive/midiclassics/<Composer>/`. Everything else under `data/` is
generated by the pipeline, and all of `data/` is gitignored.

## Setup

A conda environment named `composer-classification` with Python 3.11:

```
conda create -n composer-classification python=3.11
conda activate composer-classification
pip install -r requirements.txt
```

Everything runs from the repo root. The notebooks call the interpreter by its
absolute path; if your conda does not live at `/opt/miniconda3`, adjust the
`PYTHON` constant in the first cell of the pipeline notebooks.

## Notebooks

The numbered notebooks are the pipeline, in order. Each can regenerate its outputs
from scratch and skips work whose outputs already exist:

1. `01_data_pipeline.ipynb`: runs the extractors (MIDI copy, piano rolls, feature
   table) and creates the stratified 5 fold splits, with gate checks after each
   step. Roll extraction is the slow step on a first run.
2. `02_train_pipeline.ipynb`: trains the final model (5 folds, roughly an hour on
   Apple silicon) when `experiments/final` is missing, then presents the fold
   scores and training curves.
3. `03_model_evaluation.ipynb`: out of fold predictions, confusion matrix, per
   composer metrics, and error analysis, gated on reproducing the training scores
   exactly.

The unnumbered notebooks analyze artifacts rather than produce them: `eda.ipynb`
explores the feature table and motivates the preprocessing decisions, and
`hyperparameter_search.ipynb` presents the archived sweep that fixed the
hyperparameters (those runs are not regenerable; the notebook's committed outputs
are the record).

Every pipeline step is also a module command:

```
python -m src.data.extract_midi
python -m src.data.extract_roll
python -m src.data.extract_features
python -m src.data.make_splits
python -m src.modeling.train <run_name>
```

## Repository layout

```
src/
  data/        extractors and fold assignment
  modeling/    shared constants, dataset and loaders, model, training
  evaluation/  out of fold prediction helper
notebooks/     the five notebooks above
data/          raw archive and generated artifacts (gitignored)
experiments/   training runs: config, per epoch metrics, checkpoints (gitignored)
```

## Model

Three conv blocks (16, 32, and 64 channels) shrink a 30 second roll crop of shape
(2, 88, 300); an LSTM with 128 hidden units reads the CNN output as a sequence
over time; its final state, concatenated with the 37 feature vector, feeds a small
dense head that produces the 4 composer logits. 133,876 parameters in total.
Training uses one random crop per song per epoch, class weighted cross entropy for
the 8 to 1 class imbalance, and early stopping on held out macro-F1. Evaluation
scores whole songs by averaging window probabilities. All hyperparameters are
frozen at the sweep winners as constants in `src/modeling/`.
